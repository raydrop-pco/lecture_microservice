# マイクロサービス 101: シリーズ概要
**分散システムエンジニアリングの基礎**

*By Ray Suginaka*

## マイクロサービス 101 のめざすもの

**対象読者:** マイクロサービスの設計・開発を学ぶエンジニアおよび開発者。初級編です。
**主な目的:** アーキテクチャに対する思考の枠組み（メンタルモデル）をアップデートすること。網羅的な「How（実装手順）」の解説よりも、基本となる「Why（なぜそうするのか）」に重点を置いています。単なる手順の暗記ではなく、背後にあるメカニズムを深く理解し、それを実際の設計に応用できるようになることが大切だからです。

**マイクロサービス 101 へようこそ！**

モノリスやモジュラーモノリスの世界からマイクロサービスのエコシステムへと足を踏み入れるのは、アプリケーション開発者にとってとても楽しいことです。でも、この世界で活躍するためは、新しい技術知識の習得ではなく、**メンタルモデルの根本的な転換**が必要となります。このメンタルモデルを理解していないと、生成 AI を使ってもよいデザインはできないのです。

従来のアーキテクチャでは、安全な単一データベースのトランザクションや、瞬時に完了するメソッド呼び出しを前提としていました。ですが、分散システムでは、通信は失敗するものであり、データベースも別々にコミットされるのでデータの整合性も遅れて保たれるものです。

このシリーズでは、教科書に載っている全ての実装手法を網羅するのではなく、**「概念」と「実践的な要点」**に絞って解説します
。分散システムがなぜそのように設計されるべきなのかという「理由（Why）」にフォーカスすることで、ミドルウェアやインフラに依存しすぎず、自信を持って回復力（レジリエンス）のある設計判断ができるようになることを目指しています。

## 学習パス

* **101 シリーズ** は基礎の基礎です。マイクロサービスの開発チームに参加し、アーキテクチャの設計の基礎を学びたいすべてのエンジニア向けです。
* **201 シリーズ** は少し発展した内容で、ゼロからシステムを設計し、重要なアーキテクチャ上の意思決定を行うテックリードやシニアエンジニアを対象としています。（現在準備中）

### 各モジュール詳細

#### 101-01: 冪等性と結果整合性
**リトライしても整合性を保てますか？**
シリーズは「整合性」から始まります。リクエストがタイムアウトしてクライアントがリトライした場合、リクエストの二重処理をどう防ぎますか？本モジュールでは、冪等キー（Idempotency Key）の基礎と、結果整合性のための「処理中」という一時的なステータスの扱い方などを紹介します。

#### 101-02: レジリエンスと可観測性
**障害と「優雅に」付き合えますか？**
どんなシステムでも必ず障害は発生します。本モジュールでは、無計画なリトライの防止や、エクスポネンシャルバックオフ、ジッター、タイムアウトといったリトライを安定化させるパターンを取り上げます。同時に、問題発生時に即座に原因を特定するための可観測性（オブザーバビリティ）のコンセプトも紹介します。

#### 101-03: データの境界と所有権
**SQL の JOIN なんてなくても大丈夫ですよね？**
「システム間でデータベースを共有する」のはアンチパターンになります。サービスAでのちょっとしたスキーマ変更で、サービスBが壊れてしまうことがあります。本モジュールでは、スキーマの所有権、なぜ各サービスが個別のデータベースと対話するべきなのか、そして「Dual-Write（二重書き込み）」の失敗を避けるための「アウトボックス・パターン（Outbox Pattern）」について解説します。

#### 101-04: ワークフローとメッセージング
**バッチジョブとイベント駆動の違い、ちゃんと説明できますか？**
即時応答を待つ方式から、非同期タスクをキューで繋ぐ方式へとトレンドは移って来ました。本モジュールでは、「オーケストレーション」と「コレオグラフィ」の違いを理解し、分散処理が失敗したときでも「補償（取り消し）」アクションで一貫性を担保する Saga パターンについて深く掘り下げます。

#### 101-05: API の進化
**エコシステムを壊さずに変更を加えられますか？**
フィールド名を変更しただけでもどこかのクライアントがクラッシュする可能性があります。本モジュールでは、API スタイル（REST, gRPC, GraphQL）の比較、バージョニング戦略、そして既存クライアントを壊さないための堅牢な「Expand & Contract（拡張と縮小）」デプロイパターンについて解説します。

#### 101-06: デプロイの安全性
**リリースとデプロイを分離できていますか**
「ステージング環境では動いていたのに…」— なぜ本番デプロイは本質的にハイリスクなのでしょうか。コードのデプロイとリリースを分離することで、変更を本番環境へ安全に反映させます。ゼロダウンタイムのデプロイ（Blue-Green）や影響範囲の最小化（カナリアリリースやフィーチャーフラグ）について解説します。
